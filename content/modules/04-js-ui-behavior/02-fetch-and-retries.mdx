## Goal

Wire up a real `fetch()` call inside a `useEffect`, handle both network-level errors and HTTP error status codes, and give users a retry button that re-runs the request without reloading the page.

## Time + Difficulty

**Estimated time:** 1.5 hours
**Difficulty:** Beginner–Intermediate
The fetch API itself is simple. The complexity comes from handling all the ways a request can fail — and there are more failure modes than most beginners expect.

## Why this matters for UX Engineers

A component that fetches data but only handles the happy path is not production-ready. In the real world, networks drop, servers return 500s, rate limits kick in, and users navigate to a page before their connection is restored. A UX Engineer is responsible for the full experience — including failure.

Two things separate junior code from professional code in this area: correctly distinguishing between a network error and an HTTP error status, and giving users a clear path to recover. The `fetch` API does not throw on 404 or 500 responses — it resolves the promise. This surprises nearly every developer the first time, and forgetting to check `response.ok` is one of the most common bugs in React data-fetching code.

The retry pattern is a direct UX win: users who see a "Try again" button after a failure have a clear, low-effort path to recover. Users who see a blank screen or an untitled error page do not.

## What you will produce

A React component that:

- Fetches a list of posts from `https://jsonplaceholder.typicode.com/posts`
- Shows a loading state while the request is in flight
- Checks `response.ok` and throws if the status indicates failure
- Catches network errors separately from HTTP errors
- Shows a human-readable error message with a retry button
- Uses an `AbortController` to cancel the fetch when the component unmounts
- Allows the user to retry with a button click

## Concept overview

### fetch() does not throw on HTTP errors

This is the most important thing to understand about the Fetch API. When a server responds with a 404 or 500 status code, `fetch()` resolves the promise successfully — it considers the communication itself a success. The only time `fetch()` rejects (throws) is when the network request cannot be completed at all (DNS failure, connection refused, offline).

To catch HTTP errors, you must manually check `response.ok`:

```js
const response = await fetch(url);
if (!response.ok) {
  throw new Error(`HTTP error: ${response.status}`);
}
const data = await response.json();
```

### Network errors vs HTTP errors

| Situation | fetch() behavior | How to handle |
|-----------|-----------------|---------------|
| Server returns 200 OK | Resolves | Read `response.json()` |
| Server returns 404 or 500 | Resolves (response.ok is false) | Check `response.ok`, throw manually |
| No network connection | Rejects | Caught by `catch` block |
| Request cancelled (AbortController) | Rejects with `AbortError` | Check `err.name === "AbortError"`, ignore |

### AbortController for cleanup

When a component unmounts before a fetch completes, the fetch keeps running in the background. When it resolves, it tries to call `setState` on an unmounted component — React will warn about this, and in some cases it causes bugs. The `AbortController` API lets you cancel an in-flight request on cleanup:

```js
useEffect(() => {
  const controller = new AbortController();
  fetch(url, { signal: controller.signal });
  return () => controller.abort();
}, []);
```

When `controller.abort()` is called, `fetch()` rejects with an `AbortError`. Filter this out in your catch block so you do not set an error state for an intentional cancellation.

### Retry with a counter

The simplest retry pattern uses a counter in the dependency array of `useEffect`. Every time the counter increments, the effect re-runs and the fetch fires again:

```js
const [retryCount, setRetryCount] = useState(0);

useEffect(() => {
  // fetch logic here
}, [retryCount]);

// In error UI:
<button onClick={() => setRetryCount(c => c + 1)}>Try again</button>
```

### Exponential backoff (concept)

For automated retries (not user-initiated), it is best practice to wait progressively longer between attempts: 1 second, then 2 seconds, then 4 seconds. This prevents hammering a server that is already struggling. For this lesson, you will implement user-initiated retries only — the user decides when to retry.

### Step 1 — Create the PostList component and define types

**What:** Create a new file `PostList.tsx` with a `Post` type and a `FetchState` discriminated union.

**Where:** `src/components/PostList.tsx` in your project.

**How:**

```tsx
type Post = {
  id: number;
  title: string;
  body: string;
};

type FetchState =
  | { status: "loading" }
  | { status: "success"; data: Post[] }
  | { status: "error"; message: string };
```

**Why:** Defining the shape of the response before writing the fetch logic makes the rest of the component easier to write — Cursor will autocomplete field names based on the `Post` type.

**Check:** No TypeScript errors in Cursor. Hover over `Post` — the tooltip should show all three fields.

### Step 2 — Write the fetch inside useEffect

**What:** Add a `useEffect` that fetches from the JSONPlaceholder API, checks `response.ok`, parses the JSON, and updates state.

**Where:** Inside the `PostList` component function.

**How:**

```tsx
import { useState, useEffect } from "react";

export function PostList() {
  const [state, setState] = useState<FetchState>({ status: "loading" });
  const [retryCount, setRetryCount] = useState(0);

  useEffect(() => {
    const controller = new AbortController();

    async function loadPosts() {
      setState({ status: "loading" });
      try {
        const response = await fetch(
          "https://jsonplaceholder.typicode.com/posts?_limit=10",
          { signal: controller.signal }
        );
        if (!response.ok) {
          throw new Error(`Request failed with status ${response.status}`);
        }
        const data: Post[] = await response.json();
        setState({ status: "success", data });
      } catch (err) {
        if (err instanceof Error && err.name === "AbortError") return;
        const message =
          err instanceof Error ? err.message : "An unexpected error occurred.";
        setState({ status: "error", message });
      }
    }

    loadPosts();
    return () => controller.abort();
  }, [retryCount]);

  // render logic goes here (next steps)
  return null;
}
```

**Why:** `retryCount` in the dependency array means the effect re-runs whenever it changes. The `AbortController` ensures the fetch is cancelled if the component unmounts before the request completes. Checking `err.name === "AbortError"` prevents setting an error state for an intentional cancellation.

**Check:** Open your browser DevTools → Network tab. The request to `jsonplaceholder.typicode.com/posts` should appear when the component mounts. Confirm it returns a 200 status.

---

### Step 3 — Render each state branch

**What:** Replace the `return null` placeholder with render logic for each state.

**Where:** In the `PostList` component, below the `useEffect`.

**How:**

```tsx
  if (state.status === "loading") {
    return <p aria-live="polite">Loading posts...</p>;
  }

  if (state.status === "error") {
    return (
      <div role="alert">
        <p>Could not load posts: {state.message}</p>
        <button onClick={() => setRetryCount((c) => c + 1)}>Try again</button>
      </div>
    );
  }

  return (
    <ul>
      {state.data.map((post) => (
        <li key={post.id}>
          <strong>{post.title}</strong>
          <p>{post.body}</p>
        </li>
      ))}
    </ul>
  );
```

**Why:** The `role="alert"` attribute causes screen readers to announce the error message automatically. The `aria-live="polite"` on the loading message announces when loading begins without interrupting the user. The retry button increments `retryCount`, which triggers the `useEffect` to re-run.

**Check:** The page renders a list of 10 post titles. Open DevTools → Console. Confirm there are no React warnings or errors.

## If stuck

**The fetch URL is returning CORS errors**
JSONPlaceholder is a public API that allows CORS from any origin. If you see CORS errors, check that you copied the URL exactly: `https://jsonplaceholder.typicode.com/posts`. If you are using a different API, you may need to configure a proxy.

**The error state is not appearing when I set network to Offline**
Make sure you click the "Try again" button after setting the network to Offline, not before. The current in-flight request may still complete from browser cache. Disable caching in DevTools (Network tab → "Disable cache" checkbox) to ensure fresh requests.

**The retry button fires the useEffect multiple times**
If you used `retryCount` as a dependency and incremented it correctly, each button click should fire the effect exactly once. Check that you are not accidentally calling `setRetryCount` in more than one place.

**TypeScript complains about err in the catch block**
In TypeScript, caught errors are typed as `unknown`. You must narrow the type before accessing `.message`. Use `err instanceof Error ? err.message : "Unknown error"` as shown in Step 2.

**The loading state flashes too quickly on local dev**
On localhost, requests resolve almost instantly. Use DevTools network throttling (Slow 3G) to see the loading state for a realistic duration.

## Common mistakes

- **Not checking `response.ok`.** The Fetch API resolves on 404 and 500 responses. Without this check, HTTP errors are silently swallowed and the component tries to parse an error page as JSON.
- **Forgetting to add `AbortController`.** Unmounting during a slow request causes React warnings and can cause state to be set on a component that no longer exists.
- **Not filtering `AbortError`.** If you do not check `err.name === "AbortError"`, unmounting the component will set the error state — which causes a flash of the error UI when navigating between pages.
- **Using `catch (err: Error)` in TypeScript.** TypeScript does not allow type annotations on catch bindings — `err` is always `unknown`. Use `instanceof` to narrow.
- **Putting the fetch outside `useEffect`.** Calling `fetch()` at the top level of a component runs on every render. Always place async data fetching inside `useEffect`.
- **Forgetting `async/await` error handling.** An `async` function that throws without a `try/catch` produces an unhandled promise rejection, not a caught error.
